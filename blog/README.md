# aidemo blog — source for the engine-generated static site

The article JSON, config, style overrides, and topic map for the aidemo blog
served by GitHub Pages at **https://aidemo.top/blog/** (from the committed
`docs/blog/` output — deploy = git push). The static site is **generated by the
centralized [blog-engine](https://github.com/tandryukha/blog-engine)** (run on
demand via `npx`, pinned to `#stable`); this directory holds only the inputs.
There are no local bake/validate/hero scripts anymore — they were retired
2026-07-20 when the engine reached parity (see "Blog-engine migration (M3)"
below; git history preserves the old `scripts/*`).

The editorial design (serif reading column, drop cap, accent rules, hairline
tables, sticky TOC) is ported from the BurnWeek blog and re-tokened to the
aidemo landing palette (warm paper + terracotta) — tokens live at the top of
`templates/blog.css` (the engine reads it via `site.css`), so a brand swap is
one block. English only. No web fonts, no client framework; the only runtime JS
the engine emits is a ~25-line inline TOC scroll-spy plus a small per-block
copy-button handler on pages that contain code.

Code blocks are syntax-highlighted at bake time with shiki (`site.codeBlocks:
true` — the engine pins shiki exactly so token colors stay byte-stable): tagged
fences get inline one-light token colors re-based onto the warm panel; untagged
or unknown languages stay neutral. Each block gets a hairline container, an
uppercase language chip, and a Copy button. HTML only — the `.md` mirrors,
`llms.txt`, and `llms-full.txt` keep raw fenced code exactly as authored.

## Layout

```
blog/
  blog.config.json         machine-enforced knobs (engine reads this; see the
                           engine's docs/CONFIG-SCHEMA.md)
  data/
    articles/{slug}.json   one file per article (see schema below)
    topics.json            SEO topic/cluster map (bake tolerates absence)
    hero-subjects.json     per-slug hero subjects (one object metaphor each)
    generation-log.md      the wave lab notebook + hero-debt table
  style/
    styles.json            the matte-clay house style (engine hero lane reads it)
    header.html / footer.html   engine header/footer partials
    writer-overrides.md    voice/claims policy (STRICT layer for lane prompts)
  templates/
    blog.css               the design system (single source of CSS truth)
    banner.html            swappable CTA banner (verbatim-injected on HTML pages)
    editorial-policy.md    the E-E-A-T accountability page
  public/images/           committed WebP heroes: {slug}-400.webp, {slug}-800.webp
  scripts/daily-indexing.sh   GSC daily-indexing automation (NOT the blog build)
  ops/                     launchd template for the GSC indexing job
../docs/blog/              bake output (COMMITTED — GitHub Pages serves it)
```

## Article schema (`data/articles/{slug}.json`)

```jsonc
{
  "slug": "how-to-make-a-product-demo-video",  // kebab-case, must match filename
  "status": "draft",                            // "draft" | "published"
  "cluster": "demo-videos",                     // hub page key
  "pillar": "slug-of-pillar-or-null",           // null = this IS a pillar
  "title": "…",
  "excerpt": "120-160 chars — doubles as the article dek",
  "seoTitle": "≤60 chars, must end with ' | aidemo'",
  "seoDescription": "130-160 chars",
  "content": "markdown — starts with ## (template owns the h1); must end with '## Sources' (≥3 links) and '## FAQ' (≥2 question-form ### headings)",
  "keyTakeaways": ["3-5 bullets, ≤200 chars each, concrete numbers"],
  "hero": { "subject": "…", "scene": "…", "alt": "…", "caption": "one line that argues, not describes" },
  "tags": ["…"],
  "readingTimeMinutes": 7,
  "publishedAt": "2026-07-18",
  "updatedAt": "2026-07-18"
}
```

`topics.json` (the SEO planning lane) has shape `{clusters: {key: {title,
description}}, topics: [{slug, title, cluster, role: "pillar"|"spoke",
pillar, keywords, intent, angle, evidenceHooks, crossLinks, wave,
productMentionCap?}]}`. The bake uses it for cluster titles/descriptions, pillar
roles, and related-article crossLinks; the validator uses it as the arbiter for
roles and internal-link targets. A comparison/listicle topic that legitimately
names the product as an entry may raise its own brand-mention cap with a
per-entry `"productMentionCap": 4`.

## Commands

`blog-engine` below means `npx -y github:tandryukha/blog-engine#stable`. Run
from the **repo root** (the config path is `blog/blog.config.json`); the
`blog/package.json` `validate`/`bake`/`preview` scripts are convenience wrappers
that run the same commands from `blog/`.

```bash
# quality gate — run before every bake
blog-engine validate --config blog/blog.config.json
# + HTTP-check every Source link (citation gate: fabricated refs must not survive)
blog-engine validate --config blog/blog.config.json --resolve

# generate the static site → docs/blog (published articles only)
blog-engine bake --config blog/blog.config.json --out docs/blog
# parity check: bake to dist/ and byte-diff against the committed output
blog-engine bake --config blog/blog.config.json --verify-against docs/blog

# doctrine (served live, always version-matched): list topics, then print one
blog-engine guide
blog-engine guide WAVE-RUNBOOK      # the wave orchestration runbook
blog-engine guide AUTHORING         # the writer contract (replaces the old ops/lane-prompt.md)

# hero images (local SDXL) — see below
blog-engine heroes status --config blog/blog.config.json
blog-engine heroes render-batch --config blog/blog.config.json

# self-update the thin skill + re-pin installed.json
blog-engine skill check
blog-engine skill update --dir .

# file wave learnings upstream
blog-engine feedback

cd blog && npm run preview          # serves docs/ at http://localhost:8090/blog/
```

The bake emits: `index.html` (home), `{slug}/index.html` (articles),
`topics/{cluster}/index.html` (hubs), `sitemap.xml`, `feed.xml` (RSS 2.0,
latest 30), `images/` (copied from `public/images/`), the content-hashed
stylesheet, per-article raw-markdown mirrors (`{slug}/index.md`, advertised via
`<link rel="alternate" type="text/markdown">`), `llms.txt`, and `llms-full.txt`.
Output is deterministic — timestamps come from article fields, never the clock.
robots.txt is NOT baked here (`publish.robots: false`) — the site root
`docs/robots.txt` owns crawler policy.

## Hero images (engine SDXL lane)

Heroes are rendered locally with SDXL-Lightning by the engine's hero lane
(`blog-engine heroes …`, backed by the same $0 Mac GPU backend as the
landing-page icons), then encoded to the two WebP files the bake consumes:
`public/images/{slug}-800.webp` (article hero + og:image) and `{slug}-400.webp`
(card srcset). Both are COMMITTED — the lane runs on this machine, never in CI.

aidemo-specific conventions (config-driven, distinct from the engine default):

- **Style**: the single locked `matte-clay` house style lives in
  `style/styles.json`; `blog.config.json` sets `images.defaultStyle:
  "matte-clay"` and `images.widths: [800, 400]`. Only the subject varies, the
  style never does. Heroes render at 1216x832 (`images.heroDimensions`) — a
  native SDXL landscape bucket matching the hero `<img>` aspect in the bake.
- **Subjects** live in `data/hero-subjects.json` — one physical object metaphor
  per slug (≤12 words, no screens/text/UI, no two-object comparisons; SDXL
  renders gibberish text and cannot bind comparisons). See `blog-engine guide
  SDXL-TAXONOMY` for the full failure taxonomy.
- **Deterministic**: seed = fnv1a(slug) + `seedOffset` from hero-subjects.json.
  To re-roll one image, bump its `seedOffset` (committed → reproducible).
- Always eyeball every new image before committing (broken geometry and
  accidental text happen).

> **Known engine bug (blog-engine#15):** the hero lane hardcodes its fallback
> style to the literal `"lifestyle"` and ignores `images.defaultStyle`, so
> `blog-engine heroes …` currently crashes with `KeyError: 'lifestyle'` on
> aidemo's `matte-clay`-only styles. Non-blocking today (all 102 heroes are
> already rendered and committed), but the next hero render is blocked until the
> fix lands. Track the issue before running a new wave that needs heroes.

## Publishing flow

1. Write/edit `data/articles/{slug}.json` (status `draft`) per
   `blog-engine guide AUTHORING` + `style/writer-overrides.md`.
2. `blog-engine validate --config blog/blog.config.json --resolve` — must pass.
3. Render heroes for the new slugs (see above).
4. Flip status to `published`, re-validate, then
   `blog-engine bake --config blog/blog.config.json --out docs/blog`.
5. Commit `blog/` + `docs/blog/` together and push — GitHub Pages is the deploy
   (live in ~30-60s). Never hand-edit `docs/blog/`.

The end-to-end wave orchestration (parallel writers → ship) lives in the engine:
follow the **`aidemo-blog`** skill and `blog-engine guide WAVE-RUNBOOK`.

## Blog-engine migration (M3) — status: COMPLETED 2026-07-20

This blog was migrated onto the centralized blog-engine and the local build
machinery was retired. The engine `#stable` release (v0.2.0) reproduces the live
site byte-for-byte and gates it cleanly:

- **Bake parity**: `blog-engine bake --verify-against docs/blog` reports
  `identical (422 files)`; a scratch-dir bake `diff -r` against `docs/blog` is
  empty; re-baking into `docs/blog` is a zero-change git no-op — production is
  now engine-generated.
- **Validator**: `blog-engine validate` passes **102/102** with the config
  opt-out knobs in place (`content.cappedWords: {}`,
  `content.headingBannedWords: []`, and a `content.forbiddenStrings` that
  replaces the engine's localhost default with a cross-brand provenance guard,
  since aidemo legitimately discusses `localhost`).

What changed vs. the parked state:

1. **Engine issues #7/#8 landed** in v0.2.0 (`content.forbiddenStrings` is now
   config; per-topic `productMentionCap` is honored), and the M3 bake knobs
   (`site.codeBlocks`, `site.indexHeading`, `publish.robots`) shipped — so
   `#stable` can regenerate this site.
2. **Local machinery deleted**: `blog/scripts/{bake,validate,hero_images}.mjs`
   and `blog/ops/lane-prompt.md` (git history preserves them). `package.json`
   `validate`/`bake` now call the engine.

Ongoing use: run waves via the **`aidemo-blog`** skill; take engine updates with
`blog-engine skill update --dir .` at a wave boundary; send pipeline learnings
upstream with `blog-engine feedback` (repo-local knobs go to `blog.config.json`,
never back into deleted scripts).

## Daily GSC indexing (maintainer automation)

`scripts/daily-indexing.sh` runs daily at 10:00 via launchd
(`com.aidemo.seo-indexing`) on the maintainer's machine: it drives Google
Search Console URL Inspection for the `aidemo.top` property through a
Chrome-automation agent, submitting up to 5 never-submitted URLs and
re-checking up to 3 pending ones per day (GSC's request-indexing quota is
~10-12/day per property). The queue (`data/seo/indexing-urls.txt`) was seeded
from both sitemaps and auto-appends newly published articles on every run;
`data/seo/indexing-state.json` is the submitted/indexed ledger. Install by
substituting `__REPO__`/`__HOME__`/`__NTFY_TOPIC__` in
`ops/com.aidemo.seo-indexing.plist.template` and copying it to
`~/Library/LaunchAgents/` (the notification topic is deliberately not stored
in the repo). Delete `data/seo/gsc-property-verified` to pause the job
cleanly. First run 2026-07-20: 5/5 submitted (home, blog index, 3 topic
hubs). This automation is independent of the blog build and was untouched by
the engine migration.
